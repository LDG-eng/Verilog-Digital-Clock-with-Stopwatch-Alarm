# Experiment Setup

[cite_start]This document describes the **simulation environment**, **testbench structure**, and **verification procedure** for the Verilog Digital Clock project with **Stopwatch** and **Alarm** functionalities[cite: 5, 7].

---

## 📌 Simulation Environment

- [cite_start]**Simulation Tool:** ModelSim [cite: 117, 118]
- **Directory Structure:**
  - 📂 **`tb/`**
    - [cite_start]📜 `Top_module_tb.v` (Top-level module testbench) [cite: 17, 132]
    - [cite_start]📜 `alarm_clk_tb.v` (Alarm & current time module testbench) [cite: 15, 23]
    - [cite_start]📜 `clock_generator_tb.v` (Clock generation module testbench) [cite: 18, 109, 132]
    - [cite_start]📜 `stopwatch_tb.v` (Stopwatch module testbench) [cite: 16, 40]
- [cite_start]**RTL Source Directory:** `rtl/` [cite: 23, 40, 94, 109]
- [cite_start]**Waveform Output:** VCD (Value Change Dump) files generated for viewing in ModelSim waveform viewers[cite: 58, 118].

> [cite_start]All testbenches are **module-level**, focusing on functional verification of each block individually and integration at the top-level[cite: 14, 130].

---

## ⚡ Simulation Constraints & Parameters

[cite_start]Constraints were applied in simulation to ensure accurate timing behavior matching the project specifications[cite: 6].

- [cite_start]**Simulation Timescale:** `` `timescale 10us/1us `` [cite: 131]
- **Clock Inputs:**
  - [cite_start]Main clock: 5 KHz (`Clock_5K`) [cite: 17, 98, 112]
  - [cite_start]Derived clocks: 1 second (`Clock_1Sec`) and 1 millisecond (`Clock_1MSec`) generated by the `clock_gen` module[cite: 18, 26, 43, 114].
- [cite_start]**Reset:** Global active-high reset signal applied to initialize all modules safely [cite: 2, 26, 43, 98, 112, 138-140].
- [cite_start]**Control Signals:** System state is driven by input stimuli (`LoadTime`, `LoadAlm`, `AlarmEnable`, `Start_S`, `Stop_S`, `Reset_S`) as defined in the RTL specifications[cite: 26, 43, 98].

> **Note:** Exact FPGA/ASIC physical synthesis constraints (Area, Power) are **N/A** for this functional verification phase. [cite_start]Timing was verified against the 5KHz target[cite: 6, 98, 112].

---

## 🛠️ Verification Procedure

### 1. Module-Level Simulation
[cite_start]Each module is verified using its dedicated testbench to isolate and confirm specific behaviors [cite: 117-146]:
- [cite_start]**`clock_generator_tb.v`:** Verify 1Hz (1Sec) and 1KHz (1MSec) clock generation accuracy from the 5KHz input[cite: 114, 134].
- [cite_start]**`alarm_clk_tb.v`:** Verify timekeeping increment logic (0-59 for seconds/minutes, 1-12 for hours), AM/PM transitions, alarm trigger matching, and time/alarm manual load operations [cite: 9, 10, 26, 32-35].
- [cite_start]**`stopwatch_tb.v`:** Verify state machine transitions (start, pause, stop, reset) at millisecond resolution and ensure proper signal priority (e.g., `Reset_S` overrides `Stop_S`) [cite: 43, 49, 51-54].

### 2. Top-Level Simulation
[cite_start]`Top_module_tb.v` integrates all sub-modules and verifies system-level behavior[cite: 17, 92]:
- [cite_start]Correct mode switching (Clock ↔ Stopwatch) using the `Control` signal without signal glitches [cite: 98, 104, 151-154].
- [cite_start]Concurrent, independent operation of the Alarm and Stopwatch in the background[cite: 104].
- [cite_start]Display output correctness (`Hours`, `Mins`, `Secs`, `MSecs`, `AM_PM`, `Alarm`, `SW_State`)[cite: 104].

### 3. Waveform Analysis
- [cite_start]Simulation results are captured and analyzed to ensure all transitions meet the design specifications[cite: 58, 118, 148].
- [cite_start]**Key signals monitored:** Clocks, individual counters, AM/PM toggle, Alarm flag, and Stopwatch states [cite: 149-159].
- [cite_start]Output waveforms are securely stored in `../assets/waveforms/` and visual screenshots are available in `../assets/`[cite: 55, 116, 147].



---

> [cite_start]This setup ensures the **functional correctness** and stability of all RTL modules prior to physical synthesis, confirming suitability for FPGA/ASIC implementation flows[cite: 6, 104].
